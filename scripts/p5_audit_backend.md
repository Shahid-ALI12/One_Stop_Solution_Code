# P5 — Comprehensive Backend Bug Audit

**Task ID:** P5-AUDIT-BACKEND
**Scope:** `backend/` (FastAPI app: 12 models, 19 schemas, 19 routes, 22 services, config, db, auth, main entrypoint, requirements, pyproject, .env.example)
**Method:** Read every file end-to-end; ran `python -c "from app.main import app"` smoke test; AST-scanned for unused imports; enumerated all route registrations for duplicates and ordering issues.
**Outcome:** App imports cleanly (80 routes, 0 duplicate method+path pairs). No TODO/FIXME/XXX markers found. Found **8 HIGH**, **12 MEDIUM**, **25 LOW** bugs + 11 dead-import sites.

#### H-1 — `requirements.txt` missing auth dependencies → `pip install -r requirements.txt` produces a non-starting app
**File:** `backend/requirements.txt`
**Lines:** 1–75 (entire file)
**Description:** `pyproject.toml` declares `passlib[bcrypt]>=1.7.4`, `bcrypt<4.1`, and `python-jose[cryptography]>=3.3.0`, but `requirements.txt` (autogenerated by `uv export` but then hand-amended) omits all three. `security.py` imports `from jose import jwt` and `from passlib.context import CryptContext` at module-load time — so any environment that installs from `requirements.txt` will crash on import with `ModuleNotFoundError: No module named 'jose'` (and again for `passlib`).
**Severity:** HIGH
**Fix:** Regenerate with `uv export --format requirements-txt --no-hashes` (do not hand-edit afterward), or manually add:
```
passlib[bcrypt]>=1.7.4
bcrypt<4.1
python-jose[cryptography]>=3.3.0
```

#### H-2 — `pyproject.toml` missing `python-multipart` → upload endpoints fail at runtime under `uv sync`
**File:** `backend/pyproject.toml`
**Lines:** 7–18 (dependencies list)
**Description:** `requirements.txt` includes `python-multipart>=0.0.20` (needed by FastAPI for `UploadFile` / `Form` parsing), but `pyproject.toml` does not. Anyone who installs via `uv sync` (the documented workflow) will get a runtime error on `POST /uploads/portfolio` and `POST /uploads/resource`: `Form data requires "python-multipart"`. The error surfaces only when an upload request hits the route, not at import time, so it can slip past smoke tests.
**Severity:** HIGH
**Fix:** Add `"python-multipart>=0.0.20"` to `[project.dependencies]` in `pyproject.toml`.

#### H-3 — `GET /faqs/` exposes inactive (draft) FAQs to the public
**File:** `backend/app/routes/faqs.py`
**Lines:** 21–24
**Description:** The route signature is `def list_faqs(active_only: bool = False, ...)`. Default `False` is passed to `faq_service.list_faqs(db, only_active=False)`, which means **no** `is_active == True` filter is applied. The docstring claims "Public: returns active FAQs. Admin: pass active_only=false to see all." but there is no admin guard. Any anonymous visitor can therefore read draft / hidden FAQs by simply calling `GET /faqs/` (the default) or `GET /faqs/?active_only=false`.
**Severity:** HIGH (unintended data exposure of admin-only content)
**Fix:** Two options:
1. Default `active_only: bool = True` and document that admins must explicitly pass `?active_only=false` (still leaks to public — not recommended).
2. Use `get_optional_admin` dependency: when `active_only=False` and no admin → raise 401. Mirror the pattern already used in `ratings.py`.

#### H-5 — `SECRET_KEY` defaults to a known insecure value with no production guard
**File:** `backend/app/config.py`
**Line:** 21
**Description:** `SECRET_KEY: str = "change-me-in-production-please-use-a-long-random-string"`. The same literal appears in `.env.example` line 17. If an operator copies `.env.example` → `.env` and forgets to change it (a very common deployment mistake), the app silently runs with a publicly-known JWT signing key. Anyone can forge admin tokens. There is no Pydantic validator that raises or warns when the default is used in production.
**Severity:** HIGH
**Fix:** Add a `@field_validator("SECRET_KEY")` that raises if `value == "change-me-in-production-please-use-a-long-random-string"` and `not settings.DEBUG`, OR that enforces `len(value) >= 32` unconditionally.

#### H-6 — Default admin credentials `admin` / `admin123` with no production guard
**File:** `backend/app/config.py`
**Lines:** 30–32
**Description:** `DEFAULT_ADMIN_USERNAME: str = "admin"`, `DEFAULT_ADMIN_PASSWORD: str = "admin123"`. On first startup (or after `?force=true` reseed), `seed_service.run_seed` creates an AdminUser with these credentials. The auto-seed in `main.py` lifespan runs unconditionally on every fresh DB. In a production deployment that forgets to override these via env vars, the default admin account is wide open.
**Severity:** HIGH
**Fix:** Validator that raises on startup if `DEBUG=False` and `DEFAULT_ADMIN_PASSWORD == "admin123"`. Alternatively, refuse to auto-seed the admin if `DEBUG=False` (require explicit `POST /seed/` call).

#### H-7 — HTML injection in admin notification emails (unescaped user input)
**File:** `backend/app/services/notification_service.py`
**Lines:** 67–78 (`notify_new_enquiry`), 101–110 (`notify_new_consultation`)
**Description:** The HTML email body is built with an f-string that interpolates `enquiry.get('name', '')`, `enquiry.get('subject', '')`, `enquiry.get('message', '')`, etc. directly into `<td>` tags. A malicious visitor can submit `name = "<script>alert(document.cookie)</script>"` or `name = "<img src=x onerror=...>"`. When the admin opens the email in an HTML-rendering client, the script/executable payload fires in the admin's mailbox context. This is a stored HTML-injection → potential XSS-in-mail vulnerability. The plain-text body has the same content but is not vulnerable (text is inert).
**Severity:** HIGH
**Fix:** `import html` and wrap every user-supplied interpolation: `html.escape(enquiry.get('name', ''))`. Better: use a proper template engine (Jinja2) with autoescaping.

#### H-8 — No rate limiting on `POST /auth/login` → brute-force vulnerability
**File:** `backend/app/routes/auth.py`
**Lines:** 13–18
**Description:** The login endpoint has no rate limiting, no account lockout, no captcha. An attacker can submit unlimited username/password attempts. bcrypt verification is intentionally slow (~100 ms) which provides some defense, but a determined attacker with parallelism can still brute-force weak passwords (especially the default `admin123`).
**Severity:** HIGH
**Fix:** Add `slowapi` (or similar) rate limiter: 5 attempts / minute / IP. Also consider exponential backoff per-username and account lockout after N failures.

#### M-1 — No pagination on any list endpoint
**Files / lines:**
- `app/routes/ratings.py:28` (`list_ratings`)
- `app/routes/services.py:44` (`list_services`)
- `app/routes/enquiries.py:12` (`list_enquiries`)
- `app/routes/consultations.py:13` (`list_consultations`)
- `app/routes/resources.py:12` (`list_resources`)
- `app/routes/team_members.py:27` (`list_team_members`)
- `app/routes/faqs.py:21` (`list_faqs`)
- `app/routes/certifications.py:12` (`list_certifications`)
- `app/routes/contact_platforms.py:12` (`list_platforms`)
- `app/routes/admin_users.py:19` (`list_admins`)
- `app/routes/users.py:11` (`list_users`)

**Description:** Every list endpoint returns all rows in one response. `visits.py:list_recent_visits` is the only one that takes a `limit` (default 100). If a table grows to 10k+ rows (ratings, enquiries, visits), a single GET can produce a multi-MB JSON payload and OOM the worker. There is also no `offset` for pagination.
**Severity:** MEDIUM (DoS amplifier)
**Fix:** Add `limit: int = Query(50, ge=1, le=200)` and `offset: int = Query(0, ge=0)` to every list endpoint; apply `.limit(limit).offset(offset)` in the service layer.

#### M-2 — Consultation double-booking TOCTOU race condition
**File:** `backend/app/services/consultation_service.py`
**Lines:** 30–72
**Description:** `create_consultation` does: (1) `SELECT * FROM consultations WHERE email = ?` → list `existing`, (2) in-process `validate_slot` checks no existing booking within ±5 min, (3) `INSERT` + `COMMIT`. Two concurrent requests with the same email + slot can both pass step 2 (both see an empty `existing` list) and both commit, defeating the double-booking guard. There is no DB-level unique constraint on `(email, selected_date_time)` to backstop this.
**Severity:** MEDIUM (correctness + scheduler collision; SQLite serializes writes so the window is small, but PostgreSQL production target is fully vulnerable)
**Fix:** Add a unique index on ` Consultation(email, selected_date_time)` (or on `(email, normalized_slot_utc)`), catch `IntegrityError` and return 409. Alternatively use `SELECT ... FOR UPDATE` (PostgreSQL).

#### M-3 — `user_service.create_user` doesn't catch duplicate-email `IntegrityError`
**File:** `app/services/user_service.py`
**Lines:** 17–22
**Description:** `User.email` has `unique=True, index=True`. If a visitor submits a duplicate email, `db.commit()` raises `sqlalchemy.exc.IntegrityError`. The service does not catch it, so FastAPI returns a 500 with a raw stack trace (in debug mode). Expected behavior is 409 Conflict.
**Severity:** MEDIUM
**Fix:** Pre-check `db.query(User).filter(User.email == data.email).first()` and raise `HTTPException(409)`, or wrap commit in try/except IntegrityError → 409.

#### M-4 — `service_service.create_service` doesn't catch duplicate-slug `IntegrityError`
**File:** `app/services/service_service.py`
**Lines:** 94–110
**Description:** Same pattern as M-3. `Service.slug` is `unique=True`. Duplicate slug → uncaught `IntegrityError` → 500.
**Severity:** MEDIUM
**Fix:** Pre-check or catch `IntegrityError` → 409.

#### M-5 — `rating_stars` has no range validation
**File:** `app/schemas/rating.py`
**Lines:** 12 (`RatingCreate`), 25 (`RatingUpdate`)
**Description:** `rating_stars: int = 5` accepts any int. A visitor can submit `rating_stars: 0`, `rating_stars: -5`, or `rating_stars: 999`. The DB column has no CHECK constraint. `dashboard_service` clamps with `max(1, min(5, ...))` so charts stay sane, but the stored row is corrupted. The public `GET /ratings/` will return `rating_stars: 999` in the JSON.
**Severity:** MEDIUM
**Fix:** `rating_stars: int = Field(default=5, ge=1, le=5)` in both `RatingCreate` and `RatingUpdate`.

#### M-6 — `tz_service.parse_datetime` rejects ISO-8601 with explicit offset (`+05:00` or `Z`)
**File:** `app/services/tz_service.py`
**Lines:** 31–46 (format list), 49–55 (`_strip_tz_suffix`), 58–72 (`parse_datetime`)
**Description:** The format list includes `%Y-%m-%dT%H:%M` etc. but not `%Y-%m-%dT%H:%M:%S%z` or `%Y-%m-%dT%H:%M%z`. `_strip_tz_suffix` only strips parenthetical tokens like `(CEST)`, not trailing `+05:00` or `Z`. So `"2026-07-15T15:30:00+05:00"` and `"2026-07-15T15:30:00Z"` both fail every format and `parse_datetime` returns `None` → `validate_slot` raises `SlotValidationError(code="unparseable")` → 400 to the client. The frontend's `new Date().toISOString()` produces exactly the `Z` form, so a JS-native client would be rejected.
**Severity:** MEDIUM
**Fix:** Add ISO-8601-offset formats to `_DATETIME_FORMATS`, or first try `datetime.fromisoformat(s)` (Python 3.11+ handles `Z` natively) before falling back to the explicit format list.

#### M-7 — No global exception handler → stack traces leak in `DEBUG=True` mode
**File:** `backend/app/main.py`
**Lines:** 33–99 (`create_app`)
**Description:** No `@app.exception_handler(Exception)` is registered. With `DEBUG=True` (the default in `config.py:16`), FastAPI's default handler returns the full Python traceback in the JSON response body for any unhandled exception. This leaks internal paths, SQL fragments, and library internals to attackers.
**Severity:** MEDIUM
**Fix:** Register a catch-all handler:
```python
@app.exception_handler(Exception)
async def _unhandled(request, exc):
    logger.exception("Unhandled error")
    raise HTTPException(500, "Internal server error") from exc
```
And default `DEBUG=False`.

#### M-8 — `_client_ip` blindly trusts `X-Forwarded-For` (IP spoofing)
**File:** `backend/app/routes/visits.py`
**Lines:** 11–18
**Description:** `_client_ip` returns the first IP from `X-Forwarded-For` if the header is present, with no check that the request came through a trusted proxy. Any client can set `X-Forwarded-For: 1.2.3.4` and pollute the visits table with fake geo data, or impersonate a US/EU IP to bypass future geo-fencing.
**Severity:** MEDIUM
**Fix:** Use Starlette's `ProxyHeadersMiddleware` or only trust XFF when `request.client.host` is in a configured trusted-proxy CIDR list. Fallback to `request.client.host`.

#### M-9 — No rate limiting on any public POST endpoint
**Files:**
- `app/routes/enquiries.py:17` (`POST /enquiries/`)
- `app/routes/consultations.py:18` (`POST /consultations/`)
- `app/routes/ratings.py:72` (`POST /ratings/`)
- `app/routes/chatbot.py:23` (`POST /chatbot/`)
- `app/routes/visits.py:21` (`POST /visits/`)
- `app/routes/users.py:27` (`POST /users/`)

**Description:** None of these public endpoints have rate limiting. An attacker can flood the DB with millions of fake enquiries / consultations / visits / chatbot messages. The chatbot endpoint runs in-process DB queries on every call, making it especially cheap to DoS.
**Severity:** MEDIUM
**Fix:** Add `slowapi` limiter (e.g. 10/min/IP for write endpoints, 30/min/IP for chatbot).

#### M-10 — `visit_service._IP_CACHE` grows unbounded
**File:** `backend/app/services/visit_service.py`
**Lines:** 15, 23–47
**Description:** `_IP_CACHE: dict[str, dict] = {}` is a module-global dict with no eviction. Every unique visitor IP adds an entry that lives for the entire process lifetime. Over months of uptime with millions of unique IPs, this leaks memory until OOM.
**Severity:** MEDIUM
**Fix:** Use `cachetools.TTLCache(maxsize=10000, ttl=86400)` or `functools.lru_cache`.

#### M-11 — `DATABASE_URL` is not validated → confusing SQLAlchemy error on bad input
**File:** `backend/app/config.py`
**Line:** 24
**Description:** `DATABASE_URL: str = "sqlite:///./app.db"` has no validator. If the env var is set to a non-SQLAlchemy form (e.g. `file:/path` as in this sandbox, or `postgres://user@host` without the `+psycopg2` driver), the app fails at `create_engine(...)` with `sqlalchemy.exc.ArgumentError: Could not parse SQLAlchemy URL from given URL string` — a confusing error that doesn't tell the operator which env var to fix.
**Severity:** MEDIUM (operator experience)
**Fix:** Add `@field_validator("DATABASE_URL")` that checks the URL starts with `sqlite://`, `postgresql+`, or `mysql+`, and raises a clear `ValueError` otherwise.

#### M-12 — `seed_service.run_seed` in lifespan startup doesn't catch exceptions
**File:** `backend/app/main.py`
**Lines:** 25–29
**Description:** In the `lifespan` context, `seed_service.run_seed(db, force=False)` is called. If it raises (e.g. DB schema mismatch, integrity error from a partial prior seed), the exception propagates out of lifespan and **the app refuses to start**. There is no try/except, no logging.
**Severity:** MEDIUM (availability — startup crash)
**Fix:**
```python
try:
    seed_service.run_seed(db, force=False)
except Exception:
    logger.exception("Auto-seed failed; continuing startup")
```
